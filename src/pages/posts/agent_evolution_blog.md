# Agent 的二十年演化：它并不是大模型时代才出现的新词

这段时间，Agent 这个词突然又火了起来。

只要关注大模型应用、AI 编程、自动化工作流，几乎都会看到它：AI Agent、Browser Agent、Code Agent、Research Agent、Multi-Agent……仿佛一夜之间，所有东西都开始往 Agent 上靠。

但如果稍微往前看一点，就会发现：**Agent 根本不是一个伴随大模型才诞生的新概念。**

相反，它其实是人工智能、软件工程、机器人、分布式系统这些领域里存在了很多年的老词。只是到了今天，随着大语言模型的出现，它被重新包装、重新激活，并重新进入大众视野。

这篇文章，我想用一种尽量简单的方式，把 Agent 这二十年来的大致演化过程梳理清楚。

---

## 一、Agent 到底是什么？

很多人第一次听到 Agent，会觉得这个词很玄，好像只要系统能自动做点事，就都能叫 Agent。

某种程度上，这个感觉还真没错。

如果把各种定义压缩一下，那么 Agent 可以先粗暴理解成这样：

> **一个能够感知环境、根据目标做决策，并采取行动的实体。**

这里有三个关键词：

- 感知环境
- 做出决策
- 采取行动

只要一个系统同时具备这三点，它就已经有了 Agent 的味道。

比如：

- 一个自动巡逻的机器人，能感知周围环境，然后决定往哪走
- 一个自动下棋程序，能观察棋盘状态，然后决定下一步落子
- 一个自动处理工单的系统，能读取用户请求，然后调用不同服务完成任务
- 一个接入工具的大模型，能理解用户目标，然后搜索、读文件、写代码、调用 API

它们形态不同，但核心都一样：**不是被动等待指令后执行一条固定命令，而是会围绕目标自己做一轮“感知—决策—行动”的闭环。**

所以，Agent 从来都不是某个具体框架的专属词，也不是大模型独占的概念。它本质上是一种更高层的系统视角。

---

## 二、最早的 Agent：它更像“自治的软件实体”

如果把时间往前拨，早期的 Agent 更多出现在**多智能体系统（Multi-Agent Systems）**和**软件工程**语境里。

那时候大家谈 Agent，重点并不在“像人一样聊天”，而在于：

- 一个软件模块能不能自治
- 多个模块能不能协作
- 系统能不能在分布式环境下自组织运行

你可以把那一代 Agent 理解成一种“更聪明的对象”或者“更自治的服务单元”。

传统程序里的对象，通常是你调它一下，它执行一下。

但 Agent 讲究的是：

- 它有自己的状态
- 它能持续观察外部变化
- 它能主动做事，而不只是被动响应
- 它还能和别的 Agent 通信、协商、配合

这类思想在当年的分布式调度、制造系统、电子商务谈判系统里都很常见。

举个通俗点的例子：

假设一个系统里有“采购 Agent”“库存 Agent”“物流 Agent”“财务 Agent”。当订单进来后，它们不是简单按顺序执行一串死流程，而是可以互相交换信息、协调资源、动态决策。

这就是早期 Agent 的核心气质：**自治、协作、分布式。**

不过，这一代 Agent 虽然很有思想深度，但不够“显眼”。因为它更偏底层系统设计，不像今天的大模型产品那样容易被普通用户直接感知。

---

## 三、强化学习时代：Agent 变成“在环境里学习决策的主体”

到了后面，Agent 在另一个方向上逐渐被重新定义，那就是**强化学习（Reinforcement Learning）**。

如果你学过一点 RL，应该对这套表述很熟：

- 环境（Environment）
- 状态（State）
- 动作（Action）
- 奖励（Reward）
- Agent

在这里，Agent 的含义非常明确：

> **它是那个观察环境、采取动作、获取奖励、并试图最大化长期回报的决策者。**

注意，这和早期的软件 Agent 已经有点不一样了。

早期更强调“自治的软件实体”，而强化学习里的 Agent 更强调“决策与学习”。

比如：

- 在 Atari 游戏里，Agent 决定什么时候跳、什么时候开火
- 在围棋里，Agent 决定下一手下在哪
- 在机器人控制里，Agent 决定如何移动机械臂

这时候的 Agent 可能一句话都不会说，但它的“智能体”意味反而更强，因为它在持续和环境交互，并且围绕一个目标不断优化自身策略。

所以如果你问：**Agent 什么时候开始真正带上“智能体”那种味道？**

我觉得很大程度上就是在强化学习语境里。

因为从这时开始，Agent 不再只是“一个自治模块”，而是变成了一个能在环境中持续学习、规划和行动的主体。

---

## 四、对话系统时代：大家开始习惯“把软件包装成一个会说话的角色”

再往后，还有一条很重要但容易被忽略的支线，那就是**对话系统和聊天机器人**。

比如很多人熟悉的：

- Siri
- Alexa
- 各种客服机器人
- 任务型对话系统

严格来说，这一代系统未必都算“强 Agent”。

因为它们很多并没有特别强的自主规划能力，更多还是：

- 识别你的意图
- 抽取关键槽位
- 匹配一条流程
- 调某个已有服务
- 返回结果

比如你说“帮我订一张明天去上海的机票”，系统会提取：

- 日期：明天
- 目的地：上海
- 意图：订票

然后走订票流程。

它是有一点 Agent 味道的，但还不够强。

不过，这一阶段有一个影响非常深远：**它让人们开始习惯和“一个数字角色”对话，并通过语言调动系统能力。**

这件事看起来不起眼，实际上非常关键。

因为到了大模型时代，人们之所以能迅速接受“让一个 Agent 帮我完成任务”，很大程度上就是因为过去十多年里，大家已经被训练成习惯通过对话来操控软件能力了。

---

## 五、大模型出现后，Agent 终于把“会说话”和“会行动”缝在了一起

真正让 Agent 这个词在今天重新爆炸的，毫无疑问是大语言模型。

为什么？

因为大模型第一次把过去分散的几种能力，硬生生焊到了一起：

- 它能理解自然语言
- 它能生成计划
- 它能调用工具
- 它能根据上下文进行多步决策
- 它还能把自己的过程“解释出来”

这就很可怕了。

以前的软件 Agent 很自治，但不善于表达。

以前的强化学习 Agent 很会决策，但不会用自然语言跟人交互。

以前的聊天机器人很会说话，但很多时候不会真正做事。

而今天的大模型 Agent，第一次让这些东西同时成立：

> **它既像一个会交流的助手，又像一个能实际执行任务的系统。**

这就是为什么今天大家会觉得 Agent 非常“真”。

因为它不像以前那样只是某篇论文里的抽象定义，或者某个后台系统里的设计模式，而是真能出现在你眼前，和你对话、帮你查资料、帮你读文件、帮你写代码、帮你操作软件。

从用户视角看，这种体验就已经很接近“智能体”了。

---

## 六、为什么说今天的 Agent 不是单纯的聊天机器人？

很多人会有一个疑问：

> 这不就是“更强一点的聊天机器人”吗？

我觉得不能这么看。

今天的 Agent 和传统聊天机器人最大的区别，在于它不只是“回复你一句话”，而是能围绕目标推进任务。

一个典型的现代 Agent 往往会有这些组成部分：

### 1. 大模型
负责理解任务、拆解问题、生成下一步动作。

### 2. 工具
比如搜索、读文件、调用数据库、执行代码、访问网页、发请求。

### 3. 记忆
记住上下文、历史行为、用户偏好、长期状态。

### 4. 工作流
决定什么时候该规划，什么时候该执行，什么时候该反思，什么时候该结束。

所以现在的 Agent 更像是：

> **一个以大模型为“大脑”，以工具为“手脚”，以记忆为“经验”，以工作流为“骨架”的系统。**

这就比传统聊天机器人高了一个层次。

聊天机器人主要解决“怎么说”。

Agent 则要解决“怎么把事情做完”。

---

## 七、这两年为什么 Agent 又开始强调工具、协议和协作？

如果你继续观察近一两年的技术趋势，会发现大家谈 Agent 时，已经不只是说 Prompt、推理链、自动规划了。

现在越来越多的人开始关心：

- Tool Calling
- Function Calling
- MCP
- 多 Agent 协作
- 权限控制
- 可观测性
- 工作流编排

这说明 Agent 又进入了一个新阶段。

前两年大家热衷的是：

> “快看，大模型能自己拆任务了！”

现在大家更在意的是：

> “它能不能稳定地接入真实系统？能不能安全地工作？能不能在生产环境里跑起来？”

说白了，Agent 正从“会动的 Demo”走向“可落地的系统工程”。

这一步非常正常。

任何技术刚火的时候，大家都先迷恋能力展示；等真正要落地了，才会发现：

- 接工具怎么接
- 数据权限怎么控
- 出错怎么回滚
- 多个 Agent 怎么分工
- 上下文怎么管理
- 成本怎么压

这些东西才决定一个 Agent 系统到底是不是能用。

所以我现在对 Agent 的看法很简单：

**真正的挑战已经不再只是“让模型更聪明”，而是“让整个系统更可靠”。**

---

## 八、把这二十年的演化压成一条线

如果只想抓主线，我觉得可以记成下面这四步：

### 第一阶段：软件代理
Agent 是自治的软件实体，强调分布式、协作、主动性。

### 第二阶段：强化学习智能体
Agent 是和环境交互、通过奖励学习决策的主体。

### 第三阶段：对话代理
Agent 开始有自然语言接口，人们逐渐习惯用对话驱动软件。

### 第四阶段：LLM Agent
Agent 变成“语言理解 + 工具调用 + 记忆 + 工作流”的综合系统。

你会发现，这并不是四个完全割裂的阶段，而更像是四股力量最后合流了：

- 软件工程给了它自治与协作
- 强化学习给了它决策视角
- 对话系统给了它自然语言接口
- 大模型把这些能力统一了起来

所以今天的 Agent，其实是一个“历史合成物”。

它不是凭空诞生的，而是前面很多年积累到今天，终于找到了一个足够强的核心载体：大语言模型。

---

## 九、我的一个结论：LLM 没有发明 Agent，只是让 Agent 第一次变得全民可见

这是我写完这些内容后最强烈的一个感受。

过去很多年，Agent 其实一直都在。

只是以前它存在于：

- 学术论文里
- 机器人控制里
- 强化学习框架里
- 分布式系统设计里
- 企业后台流程里

普通人很难直接感知它。

而大模型做了一件非常重要的事：

> **它把 Agent 从“专家世界里的概念”，变成了“普通人可以直接体验的东西”。**

这也是为什么今天 Agent 会这么热。

不是因为这个概念突然被发明了，而是因为它第一次真正走到了前台。

---

## 十、写在最后

如果只看眼前的热度，很容易以为 Agent 是 AI 时代刚冒出来的新潮词。

但只要把视角稍微拉长一点，就会发现它其实是一条延续了很多年的技术脉络。

它从最初的自治软件实体，走到强化学习中的决策主体，再走到对话系统里的数字角色，最后在大模型时代重新合流，变成今天我们熟悉的样子。

所以我现在越来越觉得，理解 Agent，不能只盯着“大模型能不能自主完成任务”这一个点。

更重要的是看清它背后的历史：

- 它为什么会被提出
- 它解决过哪些问题
- 它在不同阶段到底指的是什么
- 今天的大模型又到底改变了什么

只有这样，我们才不会把 Agent 看成一个被营销反复包装的新名词，而会真正理解：

**它是一个老概念，只是在今天，终于找到了最强的表达形式。**
