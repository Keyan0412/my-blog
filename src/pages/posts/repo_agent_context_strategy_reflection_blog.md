# 大型代码仓总结不应该依赖长上下文：一次 Repo Agent 上下文策略的反思

最近在思考 repo-agent 的过程中，我又遇到了一个很典型的问题：当一个 Agent 需要总结一个大型代码仓库时，它到底应该如何处理上下文？

最直接的想法当然是，把所有相关文件尽可能都塞进模型上下文里。现在很多模型的上下文窗口越来越长，从工程直觉上看，似乎只要上下文足够大，就可以让模型一次性看到完整材料，然后直接生成全局总结。

但我越想越觉得，这个方向并不稳。

大型代码仓理解的难点，表面上是“文件很多，内容很长”，但本质上并不是单纯的 token 容量问题。真正困难的是：Agent 需要在大量文件中判断哪些信息重要，哪些只是噪声；哪些模块是主干，哪些只是辅助；哪些结论已经有证据，哪些还只是推测。

也就是说，代码仓总结不是阅读理解题，而更像一次调查。

如果把所有文件一口气塞进上下文，看起来信息最完整，但实际上很容易让模型陷入另一种混乱：它看到了很多东西，却没有形成稳定的结构。

* * *

## 一开始的直觉：上下文越完整，答案越可靠

最开始我也会自然地认为，如果要总结一个 repo，那么最理想的方式应该是让模型尽可能看到更多代码。

因为从人的角度看，材料越完整，似乎越不容易漏掉细节。尤其在代码仓分析中，很多关键逻辑会分散在不同文件里。只看 README，可能只能知道项目想做什么；只看入口文件，可能只能知道程序从哪里启动；只看核心模块，又可能忽略配置、测试和工具层的约束。

于是，一个看似合理的方案就是：

```text
读取 repo tree
  ↓
找出相关文件
  ↓
把文件内容全部放进上下文
  ↓
让模型总结项目结构
```

这个方案的问题在于，它把“上下文容量”当成了“理解能力”。

但这两者并不是一回事。

模型能看到很多内容，并不意味着它能稳定地组织这些内容。长上下文更像是一个巨大的材料堆，而不是一个已经整理好的知识结构。尤其是代码仓库这种材料，本身就包含大量重复、样板、历史遗留和边缘实现。它们都是真实信息，但并不都同样重要。

如果不先做筛选和建模，模型很容易在长上下文里抓住一些局部细节，却错过真正的主流程。

* * *

## 后来我意识到：问题不在于能不能塞进去，而在于是否应该塞进去

这也是我对长上下文逐渐产生警惕的地方。

长上下文当然有价值。它可以减少信息检索的次数，也能让模型在某些场景下直接对比更大范围的材料。但对于 repo-agent 来说，如果把它当成主要解决方案，就很容易走偏。

因为 repo-agent 的目标并不是“把仓库读完”，而是“围绕一个问题形成足够可靠的理解”。

这两件事的差别很大。

“把仓库读完”是一种材料导向的思路。它默认所有文件都应该被阅读，所有内容都可能进入上下文，最终由模型在一次调用中综合。

“围绕问题形成理解”则是一种调查导向的思路。它会先判断当前问题需要什么信息，再决定应该读哪些文件，读到什么程度，以及读完之后哪些结论值得保留下来。

前者追求完整输入，后者追求有效输入。

我现在更倾向于后者。

原因很简单：大型代码仓里真正稀缺的不是 token，而是注意力。Agent 的上下文窗口再长，也不应该被当成垃圾桶。它应该只放当前推理真正需要的内容。

* * *

## 更合理的方式：读一批，总结一批，再继续读

所以我现在更认可一种渐进式的 repo 理解方式：

```text
读取目录结构
  ↓
选择一批高价值文件
  ↓
阅读并生成局部总结
  ↓
更新 RepoProfile
  ↓
基于当前理解决定下一批要读什么
  ↓
继续调查，直到足够回答问题
```

这个过程看起来比一次性塞上下文更麻烦，但它更符合真实的软件理解过程。

人接手一个项目时，也不会从第一个文件开始顺序读到最后一个文件。通常是先看 README 和目录结构，再看入口文件，再看核心模块，再顺着调用链追下去。中间会不断形成阶段性理解，也会不断修正之前的判断。

Agent 也应该这样工作。

这里的关键不是简单地“每读几个文件就总结一次”。如果只是机械地分批读文件，效果也不会好。真正重要的是，每次总结都应该更新一个外部的、可复用的仓库画像。

我更愿意把这个东西叫做 `RepoProfile`。

它不是普通摘要，而是 Agent 对当前代码仓的阶段性理解。它应该回答的是：

```text
这个项目主要解决什么问题？
核心执行流程是什么？
重要模块分别承担什么职责？
当前已经确认了哪些事实？
还有哪些地方没有调查清楚？
哪些结论的置信度较高，哪些只是推测？
```

这样，Agent 后续继续调查时，就不需要每次都从零开始理解整个仓库，而是在已有画像的基础上进行增量修正。

* * *

## RepoProfile 的价值：让上下文从材料堆变成状态

我认为 `RepoProfile` 最大的价值，是把上下文的一部分职责外移了。

如果没有外部状态，那么模型每次调用都需要在 prompt 里重新看到大量材料。上下文既要承载原始信息，又要承载中间推理，还要承载最终回答目标。久而久之，它一定会变得臃肿。

而有了 `RepoProfile` 之后，系统可以分成三层。

第一层是即时上下文。它只包含当前问题、当前任务目标、必要的 RepoProfile 摘要、本轮要分析的文件内容，以及少量关键证据。这一层应该尽量小，因为它直接影响模型当前推理的稳定性。

第二层是运行时工作记忆。它记录当前任务已经访问过哪些文件，发现了哪些重要模块，还有哪些 open questions，下一步可能需要调查什么。这一层服务于单次任务。

第三层是持久化记忆。比如：

```text
.cache/
  repo_profile.md
  reports/
    investigation_001.md
    investigation_002.md
```

这层解决的是跨轮次、跨任务复用的问题。

如果今天已经分析过这个仓库，明天再问“planner 是怎么设计的”，Agent 不应该重新从 repo tree 开始冷启动。它应该先读已有的 `repo_profile.md`，确认目前对项目的理解，再决定是否需要补充调查。

这其实也是我认为 repo-agent 和普通长上下文调用最大的区别。

普通长上下文调用是在“读材料”。

repo-agent 应该是在“维护理解状态”。

* * *

## 但局部总结也有风险：压缩会丢失证据

当然，渐进式总结并不是没有问题。

它最大的风险是：每次总结都在压缩信息，而压缩就一定会丢失细节。如果 Agent 在早期总结中误解了某个模块，后续又一直基于这个错误画像继续推理，那么错误就会被不断放大。

所以 `RepoProfile` 不能写成一种绝对确定的百科式总结。

它应该带有证据意识。

更合理的写法应该是：

```markdown
## Main Execution Flow

当前推测主流程是：

UserQuery
→ Planner
→ Orchestrator
→ Skill
→ State Merge
→ Final Answer

Evidence:
- `src/planner.py`: `Planner.plan()` 负责把用户问题转成 plan
- `src/orchestrator.py`: `Orchestrator.run()` 负责执行 step
- `src/skills/read_file.py`: 用于读取文件并回答局部问题

Confidence: Medium

Unknown:
- 还没有确认失败重试逻辑
- 还没有确认 plan schema 是否固定
```

也就是说，它不应该只保存结论，而应该保存：

```text
结论
证据
置信度
未确认点
```

这一点很重要。

如果只有结论，那么后续 Agent 很难判断这个结论是否可靠。如果有证据和置信度，它就可以在需要时重新检查，也可以在发现冲突时更新画像。

从这个角度看，`RepoProfile` 不应该是最终报告，而应该是一个可修订的中间状态。

* * *

## Analyzer 不应该直接变成文件阅读器

结合我之前对多 Agent 分层的思考，我会更倾向于让 AnalyzerAgent 持有或读取 `RepoProfile`，但不要让它直接沉入大量文件细节。

AnalyzerAgent 的价值应该是判断：

```text
当前 RepoProfile 是否足够回答问题？
如果不够，还缺什么信息？
应该让 InvestigatorAgent 调查哪些子问题？
调查结果是否足以更新 RepoProfile？
```

它不应该亲自把十几个文件全部读完。否则它很快又会变成一个重型调查 Agent，上下文重新开始膨胀。

真正负责读文件的，应该还是 InvestigatorAgent。

但 InvestigatorAgent 也不应该漫无目的地读。它每次接到的任务应该是一个明确的子问题，例如：

```text
调查这个项目中工具调用结果是否会回到下一轮模型上下文。
```

而不是：

```text
总结 src/agents 下面所有文件。
```

前者是问题导向的调查，后者仍然是材料导向的扫描。

我现在越来越觉得，repo-agent 的核心抽象不应该是文件，而应该是问题。

文件只是证据来源。问题才是组织调查的单位。

* * *

## 这会改变整个代码仓总结流程

如果按照这个思路设计，一个大型代码仓总结不应该是：

```text
收集所有文件
  ↓
塞进上下文
  ↓
生成总结
```

而应该是：

```text
读取 repo tree
  ↓
生成初始调查计划
  ↓
优先分析 README、入口文件、配置和核心目录
  ↓
形成初始 RepoProfile
  ↓
根据 RepoProfile 中的 unknowns 继续调查
  ↓
逐步补充模块职责、执行流、数据流和关键抽象
  ↓
生成最终总结
```

这条链路看起来更长，但每一步都更可控。

更重要的是，它允许系统在中途停下来。如果用户只是想快速了解项目，那么初始 RepoProfile 可能已经够用了。如果用户要深入分析某个模块，Agent 再围绕这个模块继续调查。

这比一次性读完整个仓库更现实。

因为真实的代码理解本来就不是一次完成的。它是一个逐步收敛的过程。

* * *

## 我目前形成的判断

经过这次思考，我会把自己的判断总结成一句话：

```text
大型代码仓分析不应该依赖长上下文吞入，而应该依赖调查式阅读、增量建模和外部状态沉淀。
```

长上下文不是没用，但它不应该成为架构设计的中心。

如果系统设计的核心是“模型能塞多少”，那么整个 repo-agent 很容易退化成一个超大 prompt 生成器。它看起来简单，但不够稳定，也不容易调试。

而如果系统设计的核心是“Agent 如何维护对仓库的阶段性理解”，那么问题就会变得更加工程化：

```text
RepoProfile 如何初始化？
什么时候需要更新？
更新时如何避免覆盖旧证据？
如何表达 unknowns？
如何处理冲突结论？
如何让最终回答引用可靠证据？
```

这些问题比“上下文窗口够不够长”更重要。

因为它们决定了系统是否能持续工作，而不是只在一次 demo 中看起来有效。

* * *

## 结语

这次思考给我的一个启发是：Agent 的价值不在于把更多东西塞给模型，而在于让模型在正确的时刻看到正确的东西。

对于大型代码仓来说，真正需要的不是一个超长上下文阅读器，而是一个会持续维护理解状态的调查系统。

它应该知道自己已经理解了什么，也应该知道自己还没有理解什么。

它应该能读文件，但不应该被文件牵着走。

它应该能总结，但总结不应该变成遗忘证据的压缩。

它应该有上下文，但上下文不应该成为所有信息的唯一容器。

所以我现在更倾向于把 repo-agent 的核心能力定义为：

```text
围绕问题组织证据，并把阶段性理解沉淀为可更新的 RepoProfile。
```

这也让我对多 Agent 系统有了更清晰的判断。

多 Agent 不是为了显得复杂，而是为了把不同层级的上下文分开。高层负责判断意义，中层负责组织调查，底层负责寻找证据。只要这个边界能守住，系统就不容易失控。

而一旦所有信息又重新堆回同一个上下文里，无论它有多长，本质上都只是把混乱换了一个更大的容器而已。
