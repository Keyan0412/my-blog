最近在看 Agent 相关内容，尤其是关注新出的harness到底是啥的时候，我越来越强烈地感受到一件事：当下 Agent 的造词速度真的太快了。从早些时候的提示词工程到上下文工程，以及去年突然出现的 MCP 和 skills，这个领域似乎两三个月就出现一个新的名词，前沿领域似乎到处开花，让人摸不着头脑。不过，经过我的检索与分析以后，我发现事情并没有那么散。所谓“前沿 Agent 研究”，并不是一堆互不相关的小热点，而是几条逐层推进的主线。它们共同回答的是同一个问题，即我们到底怎样才能把一个只是会说话的大模型”，逐步变成一个能够在真实环境中长期稳定完成任务的智能体。

沿着这个问题往下看，我觉得目前真正值得长期追踪的主线，大致可以收敛到六条。它们分别对应 Agent 能力成长中的六个关键断面：工具使用与环境交互、长时程规划与任务分解、Memory 与 Context 管理、Agentic RL 与 credit assignment、Computer-use / GUI Agent，以及可靠性、评测与 Harness Engineering。

## 一、工具使用与环境交互：Agent 研究最早、也最不能跳过的底座

如果把 Agent 的发展过程想得简单一点，那么最开始的一步其实非常朴素：让模型不只是“回答”，而是“行动”。这也是为什么早期很多 Agent 系统都会围绕 Tool Use、Function Calling、ReAct 这些概念来展开。对于一个只有语言能力的模型来说，它最多只能在文本空间里做推理；而一旦它能够搜索网页、调用数据库、执行代码、访问文件系统，它才开始具备真正意义上的外部行动能力。

所以从研究脉络上看，工具使用并不只是一个工程技巧，它本质上是在回答“模型如何突破纯文本边界”这个问题。早期的做法比较直接：先让模型在推理过程中输出一个动作意图，再把这个动作映射到具体工具调用上。后来，人们慢慢意识到，真正困难的地方不只是“会不会调用工具”，而是“什么时候调用、调用哪个工具、调用之后如何理解反馈并继续决策”。

这就意味着，Tool Use 不该只被看作提示词模板的一部分，而应该被视作 Agent 决策策略的一部分。也正是在这个过程中，Agent 研究开始从“语言模型外接工具”逐渐过渡到“在环境中进行交互决策的系统”。

从今天回头看，这条线似乎已经不算最热了，因为它过于基础，很多人会下意识觉得它已经成为常识。但恰恰因为它基础，它才无法被绕过。后续你看到的规划、记忆、强化学习、GUI Agent，几乎全都建立在这一层之上。一个不能稳定感知环境、不能正确调用动作通道的 Agent，后面再复杂的设计也只是空中楼阁。

## 二、长时程规划与任务分解：从“下一步怎么做”走向“整件事怎么做完”

如果说工具使用解决的是“Agent 能不能动起来”，那么规划要解决的，就是“Agent 能不能把一件较长的事情做完”。

这是一个看起来理所当然、但其实非常棘手的问题。因为很多 Agent 在单步上并不弱，它们可以给出不错的下一步建议，也可以在局部做出看起来合理的动作。但只要任务稍微拉长，问题就会迅速暴露出来：一会儿偏离主线，一会儿重复劳动，一会儿遗忘前面已经确认过的约束条件。也就是说，Agent 的难点往往并不在于“会不会想一步”，而在于“能不能连续几十步都围绕同一个目标稳定推进”。

于是，规划与任务分解就自然成为一个独立的重要方向。最早的做法通常比较朴素：先让模型显式列出步骤，再按步骤执行。后来又出现了更复杂的层次化思路，把任务拆成目标、子目标、动作三个层次，使得 Agent 不必每一步都从头开始决策，而可以在一个较稳定的计划框架中推进。

再往后，研究重点又进一步转移。大家逐渐意识到，真正有价值的并不是“先生成一个很漂亮的计划”，而是“这个计划在执行中能不能被遵守，以及偏航之后能不能被修复”。换句话说，规划不再只是生成问题，而变成了“规划—执行—监控—重规划”的闭环问题。

我一直觉得，这条线特别值得学生关注。因为它既有很强的方法论意味，又没有脱离真实问题。规划不是那种“看起来很前沿，但很难落地”的方向，恰恰相反，它几乎天然适合作为很多 Agent 研究的中轴：往前可以接任务分解与搜索，往后可以接记忆与强化学习，横向还能和评测、可靠性分析结合起来。很多时候，一个 Agent 项目真正做出研究感，不是因为模型换得够大，而是因为你把“长任务下的规划机制”想明白了。

## 三、Memory 与 Context 管理：从“把历史塞进上下文”到“让 Agent 学会管理自己的经验”

如果只看表面，Memory 似乎只是长上下文问题的一个补丁：上下文窗口不够了，所以做检索、做摘要、做压缩，想办法把相关历史重新拿回来给模型看。但我越来越觉得，这种理解已经有些过时了。

Memory 之所以会迅速升温，并不是因为大家终于找到了更花哨的 RAG 变种，而是因为研究者逐渐意识到，长期 Agent 的核心矛盾之一，就是“系统如何持续保留、组织、调用与当前目标相关的经验”。只靠一次次把对话历史拼接回去，是撑不起真正长时任务的。一个要持续执行、不断与环境交互的 Agent，必然需要某种更主动、更结构化的记忆机制。

于是，Memory 研究的演进就很自然地分成了几个阶段。最早是原始的上下文堆叠，后来是检索式记忆，再后来开始区分不同类型的记忆，比如事件记忆、语义记忆、程序性记忆，甚至还会进一步区分用户记忆、任务记忆、环境记忆。这种变化说明，人们已经不再把记忆简单看作一个外挂数据库，而开始把它视为 Agent 内部状态的一部分。

再往前走一步，Memory 研究最吸引人的地方就出来了：问题不再只是“存什么、取什么”，而开始变成“什么时候写入、什么时候遗忘、什么时候压缩、什么时候检索”。这其实已经非常接近策略学习问题了。也就是说，Memory 不再只是一个被动的数据容器，而逐渐被看作 Agent 的主动能力。

这也是为什么很多人会觉得近一两年的 Agent 研究越来越像是在“重新发明认知架构”。因为当你认真讨论 Working Memory、Long-term Memory、Memory Update Policy 这些问题时，你已经不只是给 LLM 接几个外部模块，而是在思考：一个系统如何在长时间尺度上组织自己的经验。

从个人学习路径上说，我认为 Memory 是很适合作为切入点的一条线。它足够前沿，同时又不会像纯强化学习那样门槛过高；它既有工程落地空间，也有比较明确的研究问题；最关键的是，它与长期 Agent、个性化 Agent、陪伴型 Agent、任务型 Agent 都能自然衔接。所以如果说前沿 Agent 研究里有哪一条特别适合作为“逐步进入核心圈层”的入口，我会很愿意把票投给 Memory。

## 四、Agentic RL：从“会执行”到“会通过反馈改进自己的行为策略”

Agentic RL 这条线近来之所以热，很大程度上是因为大家逐渐发现：很多 Agent 系统虽然能工作，但它们的工作方式过于依赖启发式设计。也就是说，系统能跑起来，往往是因为研究者在外面搭了很多精巧的流程、规则与提示，而不是因为 Agent 自身真正学会了怎样在环境中做多步决策。

这样做在早期当然是必要的，因为没有这些脚手架，Agent 连基本行为都组织不起来。但当系统稍微复杂一点，问题也会随之显现：这种靠人工设计拼起来的行为模式，很难随着环境变化而稳健迁移，也很难解释为什么某些策略在一类任务中有效、在另一类任务中却迅速失效。

于是，强化学习的视角重新回到了 Agent 研究中心。不同于传统的单轮对齐问题，Agentic RL 更关注的是：当一个 Agent 在真实或半真实环境中持续行动时，它如何根据长期反馈不断调整自己的策略。这里最关键的，不再是“生成的答案好不好”，而是“整个行为序列是否高质量、哪些中间决策为最终结果做出了贡献”。

也正因为如此，credit assignment 成了一个非常核心的话题。一个几十步的任务最终失败了，到底是因为一开始的目标分解就错了，还是中途调用工具时选错了参数，还是后期因为记忆污染而偏离目标？如果不能把这种责任分配想清楚，所谓的 Agentic RL 很容易流于一个笼统口号。

所以我觉得，Agentic RL 的真正前沿性，不在于“它用了 RL”，而在于它试图把 Agent 行为真正当成一个长程决策过程来分析和优化。这条线的学术味很重，也确实更难做，因为它要求你对环境建模、奖励设计、轨迹分析、训练稳定性都有比较扎实的理解。但它也因此更接近 Agent 研究中的方法学核心。

不过，从学习策略上看，我并不建议一开始就把目标定成“做一个完整的 Agentic RL 系统”。这种说法太大了，也太容易把精力消耗在工程噪声里。更现实的路径往往是，从一个明确的子问题切入，比如记忆写入策略、工具选择策略、计划修复策略。你先把某个局部决策看成策略学习问题，再往外扩，这样反而更可能做出有质量的东西。

## 五、Computer-use 与 GUI Agent：把 Agent 从干净的 API 世界推向真实界面世界

如果说早期 Agent 研究中的环境交互，更多发生在 API、网页结构化信息、数据库等相对“干净”的接口中，那么 Computer-use 与 GUI Agent 关注的，就是一个更麻烦、也更真实的问题：Agent 如何直接在复杂的人类使用界面中完成任务。

这个方向之所以显得很吸引人，是因为它更接近日常经验。现实中的很多工作，不是调用一个格式规整的函数就能完成的，而是需要在浏览器页面上点击、在桌面应用里切换、在命令行和图形界面之间来回穿梭。也就是说，一个真正能替人做事的 Agent，不可能永远活在“接口良好、反馈清晰、动作离散”的理想环境里。

于是，GUI Agent 逐渐成为一个热点。这类研究把问题推向了一个更复杂的层面：Agent 不仅要理解语言，还要理解界面视觉元素、组件布局、可交互区域、状态变化，甚至还要在动作执行之前预估操作结果。这时候，传统的“文本输入—文本输出”范式就显得过于狭窄了。Agent 开始需要一种更接近具身系统的认知闭环：看见、判断、操作、观察变化、再判断。

也正因为环境更复杂，这条线天然更容易暴露 Agent 的真实能力。你很难靠几个精心设计的提示词，就在 GUI 场景里长期糊弄过去。因为一旦界面状态变了、元素位置变了、多个应用之间出现耦合，系统是否真的具备状态跟踪与策略调整能力，很快就会显形。

不过，这条线虽然很热，却未必适合作为所有人的第一切入口。它的门槛比较高，不只是因为要处理视觉与动作空间，还因为实验环境构建、数据采集、评测标准设计都更重。对于学生来说，更合理的路径通常不是一上来就做一个从头到尾完整的 GUI Agent，而是在已有 benchmark 或环境基础上，围绕其中某个局部问题做改进。比如做更稳的状态表示、更合理的动作选择，或者更可靠的界面变化预测。这样的切法通常更现实。

## 六、可靠性、评测与 Harness Engineering：真正决定 Agent 能否走向生产与长期研究的问题

最后这一条，在我看来是最容易被低估的。很多人谈前沿 Agent 时，更愿意谈“更自主”“更强规划”“更像人”的能力，但很少有人一开始就把注意力放在可靠性、评测与 Harness 上。可问题在于，如果一个系统不能稳定工作、不能被验证、不能被审计，那么前面那些看起来宏大的能力叙事，很可能只是漂亮演示而已。

这也是为什么近来 Harness 这个词会越来越频繁地出现。它背后的核心意思，其实不是某个神秘新方法，而是：Agent 真正可用，靠的从来不只是模型本身，而是模型外面那整套让任务可以被组织、执行、监控、恢复、验证的运行体系。一个长时任务为什么能不中途崩掉？为什么做错了还能恢复？为什么上下文超长后不会直接失忆？为什么某些高风险动作必须先经过确认？这些问题，本质上都属于 Harness 所关心的范围。

而评测问题则更进一步。随着越来越多 benchmark 出现，人们也慢慢意识到，高分并不必然代表真正的能力增长。有些 benchmark 可能被特定策略“刷分”，有些任务设计可能天然偏向某一类 Agent 架构，还有一些场景虽然在指标上看起来提升显著，但实际只是在 exploitation，而不是在 generalization。

因此，Agent 研究越往前走，就越不能只盯着“能不能做成”，而必须同时关心“为什么做成”“在什么条件下会失败”“失败是否可恢复”“指标是否真正测到了想测的能力”。

我甚至觉得，在未来几年里，真正有价值的 Agent 研究者，很可能并不只是那些最会设计自主工作流的人，而是那些最会做可靠系统、最会构造可信评测、最会分析失败机制的人。因为行业成熟到一定阶段之后，大家拼的不会再是 demo 的惊艳程度，而是系统的稳定性与可迭代性。

## 七、如果把这些主线连起来看，Agent 的演进其实是很连续的

把前面这几条线放在一起看，会发现它们之间并不是并列关系，而更像一条逐渐展开的链路。

最开始，研究关注的是 Agent 能否调用工具、与环境发生基本交互。再往前，人们发现光会动还不够，系统必须能做更长、更复杂的任务，于是规划与任务分解变得重要。任务一旦拉长，单纯依赖上下文就会失效，于是 Memory 成为核心议题。接着，人们不满足于靠手工规则组织这些模块，又开始希望 Agent 能通过环境反馈真正学会策略，所以 Agentic RL 与 credit assignment 开始升温。再往后，Agent 不再满足于在干净接口里工作，而被推向 GUI、桌面、终端等更真实的环境之中。最后，当这些能力都逐渐具备后，整个领域又不得不回到一个更务实的问题上：这样的系统到底是否可靠、是否可控、是否可评估、是否可以长期维护。

换句话说，所谓前沿 Agent 研究，并不是每年流行一个全新主题，而是在不断补齐一个智能体系统成长过程中原本缺失的关键环节。今天热的很多概念，拆开来看，其实都在围绕这条主线的不同位置展开。

## 八、如果我是以“逐步接近前沿”为目标，我会怎样安排自己的追踪路线

如果只是出于兴趣浏览，那么知道这些主线的名字、读几篇代表论文，当然已经足够。但如果目标更明确——比如希望自己未来能逐步走到前沿 Agent 研究的核心地带——那么更重要的，其实不是“全都懂一点”，而是尽快找到一条适合自己深入的主线。

在我看来，最稳妥的路径不是一上来追逐最花哨的总框架，而是先打好底层认知，再选择一条真正能持续积累的核心方向。所谓底层认知，主要是对 Tool Use、环境交互、Benchmark 范式，以及规划与记忆这类基本设计问题有足够扎实的理解。没有这一层，后面不管是做 GUI Agent 还是做 Agentic RL，都很容易陷入“看懂了很多论文名词，但自己很难提出有价值的问题”的状态。

而在具体主线上，如果一定要做取舍，我会优先关注 Memory、Planning 和 Agentic RL 这三条。它们是最像“母线”的方向，也是最容易向其他方向延伸的方向。Memory 能够自然连接长期任务和个性化系统，Planning 能够贯穿几乎所有复杂任务 Agent，Agentic RL 则指向更深层的方法学问题。相比之下，像 GUI Agent、Computer-use、Harness/Eval，虽然同样重要，但更像是这些核心方法在真实环境中的展开与检验。

所以，如果目标不是“赶一个热点”，而是“逐步形成自己在 Agent 研究中的位置”，那么与其问“现在最火的是什么”，不如问“哪些方向能够在未来两三年里持续向上生长，并且让我不断提出更深的问题”。从这个角度看，Memory、Planning、Agentic RL 的组合，几乎可以算是一条相当自然的主路径。

## 九、写在最后

回到最开始的问题：如果希望一步步接近前沿 Agent 研究领域，到底有几个可以追求的目标？

我的理解是，严格来说并不需要把世界切得太碎。真正值得长期追踪的主线，大致就是前面谈到的这六条。它们并不是互相竞争的热门词，而是在共同推动 Agent 从“会对话的模型外壳”走向“能在真实环境中持续完成复杂任务的系统”。

如果非要再收缩一步，那么其中最核心的三条，应该还是 Planning、Memory 与 Agentic RL。前者决定 Agent 能否把长任务组织起来，中者决定 Agent 能否在长时间尺度上管理经验，后者决定 Agent 能否从环境反馈中真正学会改进自己的行为。至于 Tool Use、Computer-use、Harness / Eval，更像是把这三条主线不断推向现实世界与生产约束的关键外层。

理解这一点之后，再去看那些层出不穷的新词，心态大概也会更平稳一些。因为你会知道，热词会换，包装会变，但真正的问题始终没有离开：Agent 如何感知环境、组织任务、保留经验、改进策略，并在真实世界中可靠地行动。

而所谓接近前沿，很多时候也并不意味着站在最新术语的门口，而是意味着你开始能看见这些问题之间那条连续的演进脉络。
