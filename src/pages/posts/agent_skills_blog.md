# 从 Agent Skills 谈起：它到底是能力抽象，还是过度设计？

最近在思考 Agent 架构时，我反复碰到一个问题：**skills 到底应该放在什么位置？**

一开始我很自然地把 skill 理解成一种高级 tool。因为从外观看，它们都像是“给 Agent 增加能力”的东西。tool 让 Agent 可以读文件、跑命令、搜索代码、访问 API；skill 似乎也是让 Agent 多会一点东西。但是继续想下去以后，我发现这个理解其实不够准确。

更准确地说，**tool 是一个可调用的动作，而 skill 是一种可复用的工作方式。**

tool 更像是“手里的工具”，比如 `read_file`、`search_code`、`run_shell`。它的职责应该非常清楚：给定输入，执行动作，返回结果。它不应该负责理解用户的真实目标，也不应该自己决定复杂任务的执行策略。

skill 则不一样。skill 不是简单地多封装一个函数，而是把某类任务的经验、步骤、判断规则、输出规范沉淀下来。它更像一个“任务操作手册”。当 Agent 遇到某类问题时，不再完全自由发挥，而是进入一种更受约束的工作模式。

这个区别看似简单，但我觉得它直接决定了一个 Agent 系统会不会被设计得过度复杂。

## Skill 不是固定流程，但它会约束流程

我之前曾经有一个疑问：如果 skill 是为了让 Agent 在特定场景下按照某种流程做事，那它是不是会天然损失灵活性？

答案是：会。

但问题不在于“是否损失灵活性”，而在于“损失的是哪一部分灵活性”。

一个完全自由的 Agent，看起来最聪明。用户问什么，它就自己判断怎么查、查哪里、什么时候停、怎么总结。但工程上这种自由其实很危险。它可能今天先读 README，明天先读入口文件，后天只看一个相关文件就开始总结。同一个问题，不同运行结果可能差很多。更麻烦的是，当它答错时，我们甚至很难判断问题出在哪里：是工具不好，是搜索策略不好，还是模型自己判断错了。

skill 的作用就是把这种不稳定压下来。

但好的 skill 不应该规定 Agent 每一步必须怎么走。比如一个 repo 架构分析 skill，如果写成“第一步读 README，第二步读 main.py，第三步读 config.py”，那就太死了。现实中的项目千差万别，有的 README 已经过时，有的入口根本不叫 main.py，有的核心逻辑藏在 CLI、server、package metadata 或配置文件里。

更合理的 skill 应该固定的是目标和质量标准，而不是固定路径。比如它可以要求：

- 必须找到真实入口；
- 必须找到核心模块；
- 必须解释控制流或数据流；
- 结论必须绑定代码证据；
- 证据不足时要明确标注不确定性。

至于到底先读哪个文件、要不要先做 repo map、什么时候扩大搜索范围，则应该留给 Agent 自己判断。

所以我现在更倾向于这样定义 skill：

> Skill 不是一条死流程，而是一种任务协议。它规定这个任务不能漏掉什么、不能胡说什么、什么时候该继续、什么时候可以停止，以及最后应该交付什么。

也就是说，skill 应该锁住质量下限，而不是压住能力上限。

## Skill 和 Workflow 的区别

如果把 skill 设计得太硬，它就会变成传统 workflow。

传统 workflow 适合确定性很高的任务。比如一个部署流程，必须先检查分支，再构建，再测试，再发布。这里不需要太多自由发挥，因为自由发挥反而会制造风险。

但大部分 Agent 任务不是这种形态。尤其是代码理解、debug、设计评审这类任务，过程本身带有探索性。Agent 可能需要根据当前发现不断调整方向。这里如果强行套线性流程，反而会让它变笨。

因此，我觉得 skill 最合适的形态是“半结构化流程”。

它有推荐步骤，但允许跳转；它有工具使用建议，但不绑定具体调用顺序；它有输出格式，但不要求所有问题都填同一张表。它更像一个有经验的工程师写给另一个工程师的工作指南，而不是一个自动化流水线脚本。

一个成熟的 skill，通常不只是写“第一步、第二步、第三步”。它还应该包含适用场景、完成标准、证据要求、失败处理、停止条件和常见错误。这样 Agent 才知道什么时候应该遵守流程，什么时候应该偏离流程。

## 什么时候才值得配置 Skill？

这也是我觉得最容易过度设计的地方。

不是所有 query 都值得配置一个专门的 skill。事实上，大多数简单问题根本不需要 skill。

比如用户只是问某个函数是什么意思，某个参数有什么用，某个文件大概在做什么。这些问题往往直接用 tool 读一下文件，再结合上下文回答就够了。如果为了这些问题专门做 skill，系统复杂度会增加，但收益很低。

我现在会用一个比较务实的标准判断：

> 当某类任务高频、复杂、多步、容易出错，或者答错代价较高时，才值得沉淀成 skill。

这里的重点不是“这个 query 是否专门”，而是“这个 query 背后是否有可复用的任务模式”。

比如“这个函数在哪里被调用”表面上是一个具体问题，但背后可能只是搜索引用和读几个上下文文件，不一定需要 skill。可是“这个功能请求从入口到执行是怎么流转的”，背后就可能需要找到入口、追踪 router、理解状态传递、分析多个模块协作，这就接近一个可复用的 code tracing 任务模式。

再比如“这个 bug 为什么发生”，如果只是一个简单语法错误，直接回答就行。但如果它需要复现、定位调用链、分析输入输出边界、构造根因假设、提出最小修改方案，那就很适合做成 `BugDiagnosisSkill`。

所以我更倾向于先让 Agent 自由运行一段时间，记录它实际遇到的 query 和失败情况。等某类问题反复出现，并且每次都需要相似的调查路径，再把这个路径沉淀成 skill。这样 skill 是从实践中长出来的，而不是从一开始凭空设计出来的。

## Hermes 给我的启发

我觉得 Hermes Agent 对 skill 的使用方式很值得参考。它并不是把所有问题都塞进 skill，而是把 skill 用在那些明显具有工作流特征的任务上。

比如 PR workflow、systematic debugging、TDD、代码评审、架构图生成、研究论文写作、复杂工具链使用等。这些任务有一个共同点：它们不是普通问答，而是需要 Agent 按一套比较稳定的工程方法执行一串动作。

尤其是那些和外部工具绑定的任务，skill 的价值会更明显。比如 GitHub、Jupyter、PDF、PowerPoint、Hugging Face、vLLM、Axolotl 这类系统，本身就有命令格式、认证方式、失败模式、输出规范和验证步骤。如果每次都让模型自由发挥，风险会很高。把这些经验写进 skill，就相当于给 Agent 配了一份“操作手册”。

这也让我意识到，skill 的核心场景其实不是“专门 query”，而是“专门 workflow”。

这是一个很重要的区别。

如果只是问答，Agent 可以灵活处理。如果是工作流，尤其是多步、可复用、容易失败、需要外部工具配合的工作流，skill 才真正有价值。

## 反过来看 repo-guide-agent

基于这个标准，我反而觉得我现在的 repo-guide-agent 不应该太早引入 skills。

如果当前主要场景是读 repo、解释文件、查调用、回答代码结构问题，那么核心矛盾其实不是 skill 不够，而是检索、上下文压缩、证据抽取和调用链追踪够不够好。

也就是说，现在更应该打磨的是：

让 Agent 找得准，读得对，证据给得清楚，结论不要飘。

这套能力主要依赖 tool、retrieval、ReAct 式探索和 evidence extraction，而不一定需要 skill。强行引入 skill，可能会让系统看起来更“架构化”，但实际解决不了最关键的问题。

只有当 repo-guide-agent 开始处理更完整的软件工程任务时，skill 才会变得必要。比如：

- 系统性 debug 一个项目；
- 自动生成实现计划；
- 按 TDD 增加功能；
- 做 PR review；
- 分析一次重构的风险；
- 自动维护项目文档；
- 生成架构图并持续更新。

这些任务已经不只是“回答代码问题”，而是“执行工程工作流”。这时候 skill 才有明显价值。

因此，我现在对 repo-guide-agent 的判断是：不要为了追求架构完整而提前引入 skills。先把底层代码理解能力做好，等任务复杂度真的上来，再把反复出现的工作流沉淀成 skill。

## 我对 Skill 的最终理解

经过这轮思考，我现在会把 Agent 系统里的几层关系这样理解：

tool 是原子动作。  
skill 是任务方法。  
agent 是调度与决策系统。  
workflow 是更硬的执行流程。  
memory 则保存长期经验和上下文。

skill 处在 tool 和 workflow 之间。它比 tool 更高层，因为它包含任务策略；但它又不应该像 workflow 那样死板，因为它仍然要保留 LLM 的探索能力。

所以我认为一个好的 skill 应该具备几个特征：

第一，它对应的是任务模式，而不是具体对象。不要做 `ExplainRouterSkill`、`ExplainPlannerSkill` 这种太细的东西，而应该做 `CodeTraceSkill`、`BugDiagnosisSkill`、`DesignReviewSkill` 这种可复用的任务能力。

第二，它固定的是质量标准，而不是具体路线。它应该规定必须有证据、必须检查哪些风险、必须在什么情况下停止，而不是规定永远先读哪个文件。

第三，它应该从真实使用中沉淀出来，而不是一开始就设计一大堆。skill 太多以后会变成维护负担，也会让 Agent 的行为变得僵硬。

第四，它更适合复杂工作流，而不是普通问答。只要 tool 和模型本身可以稳定解决的问题，就不必上 skill。

## 结语

我现在越来越觉得，Agent 架构设计最容易犯的错，就是看到一个概念很好，就想立刻把它塞进自己的系统里。

skill 确实是一个很有价值的抽象，但它不是所有 Agent 都必须拥有的模块。它真正解决的问题，是把复杂任务中反复出现的有效工作方式沉淀下来，让 Agent 下次不要再从零开始摸索。

所以，skill 的出现应该有一个前提：系统已经遇到了足够复杂、足够高频、足够容易失败的任务。

在那之前，最重要的不是增加 skill，而是让 Agent 的基本能力足够扎实：能正确搜索，能读懂上下文，能引用证据，能承认不确定性，能在信息不足时继续调查。

等这些基础做好以后，skill 才不是装饰性的架构概念，而会自然变成系统能力增长后的沉淀物。
