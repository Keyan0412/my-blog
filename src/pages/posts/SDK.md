很多人在第一次接触 SDK（Software Development Kit）时，会下意识把它理解为“一个工具集合”或者“一个封装好的代码库”。这个理解不能说错，但如果停在这里，基本上永远做不出一个真正能被别人使用的 SDK。

更接近现实的说法是：**SDK 不是代码形态的变化，而是思维方式的变化。**

---

## 一、SDK 的本质，不是“写给自己”的代码

大多数项目，从诞生那一刻起就是“自用”的。

你写一个脚本、一个 Agent、一个推荐系统，默认前提是：  
你知道输入是什么，知道输出长什么样，知道什么时候会出错，甚至知道哪里写得很糙。

但 SDK 完全不同。

它的前提是：

> 使用你这段代码的人，对你的内部实现一无所知，而且也不应该关心。

这就导致一个关键变化：

- 你不能再依赖“上下文理解”
- 你不能默认“使用者会读源码”
- 你不能把复杂性暴露出去

换句话说，**SDK 是把“能力”变成“接口”的过程**。

---

## 二、为什么大多数所谓 SDK 都不好用

如果你用过一些开源 SDK，应该有这种感觉：

- 文档模糊，看了等于没看
- API 参数一堆，但不知道该怎么组合
- 报错信息毫无意义
- 改个版本就全炸

这些问题的根源，其实只有一个：

> 开发者在用“写项目”的方式写 SDK

写项目时，核心关注点是“能跑”；  
写 SDK 时，核心关注点是“别人能不能稳定地用”。

这两者之间，有一个非常大的鸿沟。

---

> 用户会怎么调用你？
