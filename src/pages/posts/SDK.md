最近在写个人项目时接触到了 SDK（Software Development Kit）词，经过详细了解后发现这个词还有些名堂，因此准备出一期博客专门介绍一下。
这个词听起来像是"一个工具集合"或者"一个封装好的代码库"，但它的定义好像并不止于此。

------------------------------------------------------------------------

## 一、SDK 的本质

大多数项目，从诞生那一刻起就是"自用"的。你写一个脚本、一个项目、或是一个系统，默认前提是：\
你知道输入是什么，知道输出长什么样，知道什么时候会出错，甚至知道哪里写得很糙。

但 SDK 完全不同。 它的前提是：

> 使用你这段代码的人，对你的内部实现一无所知，而且也不应该关心。

这就导致一个关键变化：

-   你不能再依赖自己的理解
-   你不能默认使用者会读源码
-   你不能把复杂性暴露出去

换句话说，**SDK 是把"实现"变成"接口"的过程**。

------------------------------------------------------------------------

## 二、SDK 开发的第一步：先设计使用方式

一个常见误区是，写一个项目一上来就开始写实现。 但在 SDK 里，实现反而不是最重要的部分。真正重要的是用户会怎么调用这个工具。

比如做一个推荐系统，如果是普通项目，可能会这样写：

``` python
results = pipeline.run(query)
```

但如果是 SDK，则必须思考更多：

-   用户是否需要初始化客户端？
-   是否需要认证？
-   是否支持配置超时、重试？
-   输出是否结构化？
-   是否支持扩展参数？

于是接口会变成这样：

``` python
client = RecommendationClient(api_key="xxx")
results = client.recommend("LLM memory", top_k=5)
```

------------------------------------------------------------------------

## 三、隐藏复杂性，是 SDK 最核心的能力

一个成熟 SDK 的标志是使用逻辑清晰。

你内部可能有：

-   需要调用的外部模型
-   复杂的代码逻辑
-   底层的资源管理

但这些东西，用户都不应该看到。 用户只关心一件事：

> 我给你一个输入，你给我一个结果。

------------------------------------------------------------------------

## 四、SDK 的结构，其实是一种分层控制

如果把一个 SDK 拆开来看，通常会有一种很明显的分层：

最外层是 client，负责对外暴露能力；\
中间是 core，负责真正的逻辑；\
底层是 utils / models / errors，提供支撑。

关键在于：

> client 层必须极简，而 core 层可以复杂。

------------------------------------------------------------------------

## 五、错误处理：决定 SDK 是否可用的关键

在普通项目里，报错往往只是为了调试；\
但在 SDK 里，报错本身就是接口的一部分。

如果你只是简单地：

``` python
raise Exception("something wrong")
```

那使用者几乎无法处理这个错误。

------------------------------------------------------------------------

## 六、版本控制：SDK 的"稳定性承诺"

SDK 一旦被使用，就不再只是你的代码。

每一次修改，都会影响别人。

这也是为什么 SDK 必须严格遵循版本规范。

------------------------------------------------------------------------

## 结语

SDK 这个词听起来像是"工程细节"，但实际上，它更接近一种产品思维。 它不再只是解决问题，而是在思考：

-   别人如何接入
-   如何保证稳定
-   如何隐藏复杂性
-   如何长期演化
